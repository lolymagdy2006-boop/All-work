Introduction to Exchange Formats: JSON and Database Backup/Migration

Introduction to exchange formats ( CSV , XML , JSON ...) and database backup / migration - Part 3

This final 90-minute session focuses on JSON (JavaScript Object Notation), the reigning champion of modern web data exchange. While CSV is the "past" and XML is the "enterprise giant," JSON is the "language of the internet."

Part 1: Why JSON Won the Web (20 Minutes)

In the early 2000s, XML was the standard. However, as web applications became more interactive, developers needed something faster and easier to use with JavaScript.

The Origins: Derived from JavaScript, but language-independent. Almost every programming language can parse JSON today.

The "Weight" Factor: JSON is much less "wordy" than XML. There are no closing tags (e.g., ), which reduces file size and saves bandwidth.

Key Philosophy: Data is organized into Key-Value Pairs and Arrays.

Part 2: JSON Anatomy & Data Types (30 Minutes)

JSON is built on two primary structures: Objects (curly braces {}) and Arrays (square brackets []).

1. The Building Blocks

Unlike CSV, JSON supports data types natively:

Strings: "John" (always double quotes)

Numbers: 25 (no quotes)

Booleans: true / false

Null: null

Objects/Arrays: For nesting.

2. Syntax Rules (The "Must-Follows")

JSON is strict, though slightly more forgiving than XML:

Keys must be strings in double quotes.

No trailing commas (a common error in migration scripts!).

The entire file must be a single JSON value (usually an object or an array).

Part 3: JSON in Databases & Migrations (40 Minutes)

In the past, databases were strictly relational (rows and columns). Today, we live in a "Polyglot" world where JSON is a first-class citizen.

1. NoSQL and Document Databases

Databases like MongoDB store data directly as JSON-like documents (BSON).

Migration Advantage: You don't need to define a strict schema before moving data. This is called Schema-on-Read.

Use Case: Ideal for rapid development or data where the structure changes often.

2. JSON in Relational DBs (PostgreSQL/MySQL)

Modern SQL databases now have a JSONB data type.

Hybrid Migration: You can move 90% of your data into standard columns and "dump" the unpredictable 10% into a JSON column.

Extraction: It is very common to export SQL query results directly as a JSON array for an API.

3. The Migration Trap: "Deep Nesting"

While JSON handles hierarchy beautifully, nesting data 10 levels deep makes it very hard to query.

Lecture Tip: During migration, "flatten" your JSON where possible to keep it performant.

Summary: The "Big Three" Comparison

Feature CSV XML JSON
Simplicity High Low Medium
Nesting No Yes (Deep) Yes (Flexible)
Data Types No Limited Yes (Strong)
Usage Spreadsheets/Legacy Enterprise/Banking Web APIs/Mobile Apps

Cheat Sheet comparing all three formats (CSV, XML, JSON)

Here is the comprehensive Data Exchange Format Cheat Sheet. This is designed to be the ultimate reference for your students to use during lab work or for quick study.

Feature CSV (Comma Separated) XML (Extensible Markup) JSON (JS Object Notation)
Structure Flat (Rows & Columns) Hierarchical (Tree) Hierarchical (Key-Value)
Self-Describing No (Header only) Yes (Tags) Yes (Keys)
Data Types None (All are Strings) Limited (Mostly Strings) Native (String, Num, Bool, Null)
File Size Smallest (Efficient) Largest (Verbose) Medium (Balanced)
Strictness Low (Very loose) High (Must be Well-Formed) Medium (Strict Syntax)
Best For Data Science, Bulk Imports Enterprise, Banking, Legacy Web APIs, Mobile Apps, NoSQL

When to Use Which?

Use CSV when...

You are moving massive amounts of simple tabular data (e.g., millions of log entries).

You need to open the data in Excel or a basic text editor.

Bandwidth and storage space are your #1 priority.

Use XML when...

You need strict validation (XSD) before the data even touches the database.

You are working in "High-Stakes" environments (Banking, Legal, Medical).

You need to store complex documents with many nested layers and metadata.

Use JSON when...

You are building a Web or Mobile Application.

You are using a NoSQL database (like MongoDB) or a modern SQL JSON column.

You want a balance between human readability and machine efficiency.

Glance Example

Since you've tackled the theory and the testing, let's get practical! Here is a Python "Master Script" that demonstrates how to convert a single data record between all three formats. This is the ultimate tool for understanding how a computer actually "sees" these differences.

The Conversion Script

This script takes a simple list of dictionaries (representative of database rows) and exports it into CSV, XML, and JSON.

import csv
import json
import xml.etree.ElementTree as ET

# 1. THE DATA (The "Source")
data = [
    {"id": "1", "name": "Gemini", "type": "AI", "status": "Active"},
    {"id": "2", "name": "Alpha", "type": "Bot", "status": "Offline"}
]

# 2. TO CSV (The Flat Export)
with open('data.csv', 'w', newline='') as f:
    writer = csv.DictWriter(f, fieldnames=data[0].keys())
    writer.writeheader()
    writer.writerows(data)

# 3. TO JSON (The New Export)
with open('data.json', 'w') as f:
    json.dump(data, f, indent=4)

# 4. TO XML (The Hierarchical Export)
root = ET.Element("SystemLog")
for entry in data:
    item = ET.SubElement(root, "entry", id=entry['id'])
    for key, val in entry.items():
        if key != 'id':
            child = ET.SubElement(item, key)
            child.text = val
tree = ET.ElementTree(root)
tree.write('data.xml', encoding='utf-8', xml_declaration=True)

print("Migration complete! Files generated: data.csv, data.json, data.xml")

What to Notice in the Output:

CSV: You'll see a clean table, but notice that the "id" is just a string in a column.

JSON: You'll see the square brackets [] denoting an array of objects‚Äîready to be sent to a mobile app.

XML: Notice how the script used the id as an attribute inside the tag, while name and type became nested elements.
